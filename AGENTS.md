# Codex作業ルール

このファイルはリポジトリ全体に適用する。Skillsには依存せず、Codexは変更タスクで以下の手順を標準ワークフローとして使用する。

## 適用範囲

- ユーザーが実装、修正、追加、削除を依頼した場合は、原則として「計画 → 開発 → 検証 → Push → PR → CI確認」まで完了する。
- 説明、調査、レビュー、診断だけの依頼では、ファイル変更、コミット、Push、PR作成を行わない。
- ユーザーが「ローカル変更だけ」「Push不要」など範囲を指定した場合は、その指示を優先する。
- PRのマージはユーザーが明示的に依頼した場合だけ行う。

## 1. 計画と着手前確認

1. `git status --short`、現在のブランチ、直近の履歴を確認する。
2. 関連する設定、スクリプト、CI、READMEを読み、リポジトリから分かる事実を先に調査する。
3. 作業内容、影響範囲、検証方法を短くユーザーへ共有する。複雑な作業では進捗管理用のplanを使用する。
4. `plan.md`、調査メモ、インベントリなどの一時文書は、ユーザーが明示的に求めない限りコミットしない。
5. 未追跡ファイルや無関係な変更はユーザーのものとして扱い、変更・ステージ・削除しない。

## 2. ブランチ運用

- `main`へ直接コミットまたはPushしない。
- 作業開始時にPRがマージ済みか確認し、最新の`origin/main`をfast-forwardで取得してから専用ブランチを作る。
- ブランチ名は変更内容に合わせ、`feat/`、`fix/`、`refactor/`、`test/`、`docs/`、`chore/`のいずれかを使用する。
- 既存の未マージPRに対する追加修正なら、そのPRのブランチを継続して使う。
- 作業中にPRがマージされてmainへ戻っていた場合は、最新mainから新しいブランチを作り、マージ済みブランチへ追加コミットしない。

## 3. 開発

- 変更は依頼された目的に必要な最小範囲に限定する。
- 依存関係は既存の宣言元を使う。
  - Homebrew: `Brewfile`
  - Ubuntu固有パッケージ: `packages/apt.txt`
  - Bun管理LSP: `packages/bun-lsp/package.json`と`bun.lock`
  - Neovim/Mason/Sheldon: 各設定ファイル
- macOSとUbuntu/WSLの両方を考慮し、アーキテクチャやユーザー固有の絶対パスを埋め込まない。
- 動作変更に対応するテストまたはスモークテストを追加・更新する。
- ユーザー向けの利用方法や前提条件が変わる場合は、日本語の`README.md`も更新する。

## 4. ローカル検証

変更内容に応じて必要な検証をすべて実行する。最低限、以下を確認する。

```sh
git diff --check
pre-commit run --all-files
```

追加の基準:

- Bash変更: `bash -n`とShellCheck
- Zsh変更: `zsh -n`（`.zshrc`はShellCheck対象外）
- GitHub Actions変更: actionlint
- JSON/YAML変更: pre-commitの構文検査
- Neovim変更: ヘッドレス起動と関連LSP・プラグインの初期化確認
- セットアップ動作変更: `scripts/smoke-test.sh`を更新し、実行可能な環境では実行する

ローカルでフルセットアップするとユーザー環境を破壊する可能性があるため、`./setup.sh --force`やパッケージ削除は検証目的だけで実行しない。完全セットアップはGitHub Actionsの使い捨てrunnerで検証する。

検証が環境制約で実行できない場合は、理由と未検証項目を明記する。検査失敗を無視してコミットしない。

## 5. コミットとPush

1. `git diff`と`git status`で最終差分を確認する。
2. 今回の変更ファイルだけを明示的にステージする。`git add .`は使用しない。
3. Conventional Commits形式で、変更目的が分かるコミットを作成する。
4. pre-commit hookの結果を確認する。hookがファイルを修正した場合は再検証してからコミットする。
5. 作業ブランチを`origin`へPushする。

## 6. Pull Request

- 同じheadブランチの未マージPRがなければ、`main`向けPRを作成する。重複PRを作らない。
- PR本文には最低限、以下を記載する。
  - 変更の目的と概要
  - 主な実装内容
  - 実行したローカル検証
  - 制約、未検証事項、手動確認が必要な点
- ローカル専用文書をPRに含める必要がある場合は、ファイルをコミットせずPR本文へ記載する。

## 7. CI確認と修正ループ

1. PR作成またはPush後、macOSとUbuntuの全チェックが完了するまで確認する。
2. 失敗した場合は、GitHub Actionsの失敗ジョブログを取得し、推測ではなくログに基づいて原因を特定する。
3. 修正、ローカル検証、コミット、Pushを行い、CIを再確認する。
4. macOSとUbuntuの両方が成功するまでこのループを続ける。
5. 警告と失敗を区別する。警告が将来の障害につながる場合はユーザーへ報告する。

## 8. 完了報告

最終報告には以下を簡潔に含める。

- 実装結果
- 主要な変更ファイルまたは動作
- 実行した検証と結果
- コミットID
- PR URL
- macOS／Ubuntu CIの結果

CIが未完了または未実行の状態を「完了」と報告しない。
